## 0. 문서 목적

이 문서는 **투어게더(Tourgether) 플랫폼에 NFT 기반 멤버십/프리미엄 시스템을 도입하기 위한 기획/설계 초안**이다.

- 개발자 입장에서는:  
  - 전체 구조를 이해하고, 어떤 API/DB/외부 연동이 필요한지 감을 잡기 위한 문서
- 투자자/이해관계자 입장에서는:  
  - “이 NFT가 단순 투기용이 아니라, 서비스 안에서 어떤 가치를 만드는지”를 빠르게 이해하기 위한 문서

이 문서는 **기술 스택 예시는 Replit + Web3(예: Polygon)**을 전제로 작성한다.

## 1. 서비스 컨셉 요약

### 1.1 한 줄 요약

> **신뢰를 쌓은 여행제공자(가이드/셀러)만, 자신의 이름이 걸린 “클럽 NFT”를 발행할 수 있고,  
>  이 NFT를 가진 고객은 투어게더 안에서 한 단계 높은 서비스/경험/위상을 누리게 된다.  
>  NFT는 OpenSea 등에서 자유롭게 거래되며, 플랫폼과 발행자는 지속적인 로열티 수익을 기대할 수 있다.**

### 1.2 핵심 개념

- **플랫폼 (투어게더)**  
  - NFT를 “직접” 발행하는 주체  
  - 누가 발행할 수 있는지 심사/승인  
  - 발행 수수료, 2차 거래 로열티 중 일부를 수익으로 가져감  
  - NFT 보유량에 따라 서비스 내 권한/경험을 달리 제공

- **크리에이터(가이드/셀러)**  
  - 일정한 신뢰도/리뷰/실적을 달성한 후  
  - “본인 전용 NFT 컬렉션” 발행을 신청할 수 있음  
  - 자신의 NFT를 상품 옵션, 멤버십 패스, 팬 뱃지 등으로 활용  
  - 판매/거래에 따른 일정 로열티를 지속적으로 얻음

- **홀더(일반 유저/팬)**  
  - 크리에이터의 NFT를 구매/획득/보유하는 사용자  
  - NFT 보유 개수 및 기간에 따라  
    - 우선 예약, 비밀 지도, 가격 고정, 오프라인 모임 등  
      다양한 프리미엄 경험을 얻음

- **NFT (클럽/멤버십 토큰)**  
  - 블록체인 상에 기록되는, “크리에이터 전용 디지털 멤버십 카드/트로피”  
  - 발행량·기간·자격 요건이 제한되어 있어 희소성이 있음  
  - 보유 기간이나 사용 이력에 따라 **외관/속성이 변하는 Dynamic NFT**로 설계

## 2. NFT 상품 구조

### 2.1 NFT 유형 (Creator Membership NFT)

- 기본 유형: **“Creator Membership NFT”**
- 각 크리에이터는 **최대 N개(예: 1~2개)** 의 컬렉션만 운영 가능
- 예시:
  - `Tourgether | Guide_A Club Pass`
  - `Tourgether | Seller_B VIP Relic`

### 2.2 NFT 기본 속성 (메타데이터 개념)

각 NFT는 다음과 같은 공통 속성을 가진다. (실제 온체인/오프체인 메타데이터로 구현)

- `platform`: `"Tourgether"`
- `creator_id`: 플랫폼 내부 크리에이터 식별자
- `creator_wallet`: 크리에이터 지갑 주소
- `collection_name`: 컬렉션 명칭
- `token_type`: `creator_membership`
- `season`: 시즌/기수 정보 (예: `2025_Q1`)
- `base_image_url`: 기본 이미지
- `dynamic_state`: 현재 등급/단계 (예: `bronze`, `silver`, `gold`)
- `benefit_profile_id`: 어떤 혜택 패키지를 참조하는지
- `supply_cap`: 총 발행 한도 (예: 300개)
- `mint_start_at`, `mint_end_at`: 민팅 가능한 기간

### 2.3 희소성 설계

1. **발행량 제한**
   - 컬렉션별 `supply_cap` 정의 (예: 100~500개 범위)
   - 일정 수량이 소진되면 신규 민팅 불가

2. **기간 제한**
   - `mint_start_at` ~ `mint_end_at` 사이에만 민팅 가능  
   - 시즌 종료 후, 해당 시즌 NFT는 “완전 종결된 레거시 아이템”이 됨

3. **Dynamic NFT (외관 변화)**
   - 조건 예시:
     - 보유 기간 (예: 3개월, 6개월, 1년)
     - 해당 크리에이터와의 거래 횟수 (예: 1회, 5회, 10회)
     - 컬렉션 완판 여부
   - 조건 만족 시:
     - 이미지/배경/아이콘/타이틀이 변경  
     - 예: 브론즈 → 실버 → 골드 반지로 변신

## 3. 역할 및 사용자 시나리오

### 3.1 크리에이터(가이드/셀러) 플로우

1. **자격 획득**
   - 플랫폼 내부 “신뢰도 점수” 기준 충족:
     - 최소 거래 수, 평균 평점, 분쟁률, KYC 등

2. **NFT 발행 신청**
   - 마이페이지에서 `NFT 멤버십 만들기` 클릭
   - 다음 정보를 입력:
     - 컬렉션 이름
     - 시즌/기간
     - 발행량 상한
     - 혜택 구성 (아래 6장에 예시)
     - 기본 이미지 및 스타일

3. **플랫폼 심사**
   - 플랫폼 운영팀/알고리즘이:
     - 설명의 적절성
     - 과장/허위/투자성 표현 여부
     - 혜택의 실현 가능성
   - 통과 시 “승인”

4. **발행 수수료 결제**
   - 승인 후, 크리에이터는 **발행 수수료**를 플랫폼에 결제
     - 예: 100~300 USDT 상당 (구체 금액 추후 결정)
   - 이 수수료는:
     - 스팸/가치 없는 컬렉션 난발 방지
     - 플랫폼 기본 수익원

5. **컬렉션 생성 및 OpenSea 노출**
   - 플랫폼이 스마트컨트랙트 호출 → 컬렉션 생성
   - OpenSea에서 `Tourgether | {CreatorName}` 컬렉션으로 노출

6. **상품/서비스와 연동**
   - 크리에이터는 자신의 투어/상품 옵션에서
     - “NFT 홀더 전용 옵션”
     - “NFT 보유자 할인/우선권 옵션”
   - 을 설정 가능

### 3.2 홀더(일반 유저/팬) 플로우

1. **투어게더에서 크리에이터 발견**
   - 투어/상품 상세 페이지에서
     - “이 크리에이터의 NFT 멤버십 소개” 섹션 확인

2. **NFT 획득 방법**
   - (1) 투어게더 내 민팅/판매 페이지에서 직접 구매
   - (2) OpenSea에서 2차 거래로 구매
   - (3) 특정 조건 달성 시(리뷰, 반복 이용 등) 보상으로 획득

3. **NFT 보유 지갑 연결**
   - 투어게더 계정에 본인 지갑 연결
   - 플랫폼이 온체인 조회로 **보유 개수/상태**를 확인

4. **티어/권한 부여**
   - NFT 개수 + 상태에 따라:
     - 브론즈/실버/레전드 티어 결정
   - 각 티어별로 다른 메뉴/기능/혜택 오픈

### 3.3 플랫폼 운영자 플로우

- 크리에이터 자격 관리
- NFT 발행 심사/승인
- 혜택 설계 가이드 제공
- 규제/리스크 관련 문구 및 이용약관 관리
- 블록체인 노드/메타데이터 서버 안정성/보안 관리

## 4. 비즈니스 룰

### 4.1 발행 자격 요건 (예시)

- 지난 12개월 내 완료된 거래 수 ≥ 50건  
- 평균 평점 ≥ 4.5/5.0  
- 분쟁/환불률 ≤ 5%  
- 신원 인증(KYC) 완료  
- 약관 동의 및 NFT 운영 정책 서약

### 4.2 발행 수수료 구조 (예시)

- **기본 발행 수수료**: 1회 컬렉션 생성 시 고정 수수료
- **유지/업데이트 수수료**: Dynamic NFT 기능 등 추가 기능 사용 시 옵션 수수료
- **수익 분배 구조(2차 거래 로열티)**:
  - 2차 거래 시, OpenSea 등에서 로열티가 발생할 경우:
    - 크리에이터: 예) 70%
    - 플랫폼: 예) 30%
  - 단, 실제 로열티 지급은 마켓별 정책에 따라 달라질 수 있음 (강제 불가할 수 있음)

### 4.3 NFT 혜택 종류 (프리미엄 가치 예시)

아래 항목은 “이 중 일부를 선택/조합”하는 형태로 설계할 수 있음.

1. **우선 예약권 (Priority Booking)**
   - 인기 가이드 일정 예약 시,  
     NFT 홀더에게 24시간 먼저 예약 창 오픈

2. **가격 고정/할인 (Price Lock)**
   - 연간 물가 상승과 관계 없이  
     NFT 홀더는 특정 기간 동안 동일 가격으로 예약 가능

3. **비밀 지도/코스 열람 (Hidden Map Access)**
   - 일반 유저에게는 보이지 않는  
     로컬 밀착 코스/POI 리스트에 접근 가능

4. **막차 구조권 (Last-minute SOS)**
   - 예약 취소/노쇼 발생 시,  
     NFT 홀더에게 우선적으로 대체 슬롯 제안

5. **오프라인 클럽 모임 (Owner Trip / Salon)**
   - 연 1~2회, NFT 홀더 전용 소규모 오프라인 모임/투어

6. **플랫폼 내 노출 강화 (Boost)**
   - NFT 홀더의 후기/콘텐츠는 피드/지도에서 살짝 강조 표시  
   - 크리에이터 입장에선 “좋은 고객층”을 한눈에 파악 가능

7. **라이트 거버넌스 권한**
   - 새로운 도시/테마 출시 우선순위 투표  
   - 특정 이벤트/기능 도입에 대한 의견 반영 (비법적 자문 수준)

> **주의:**  
> - NFT 홀더에게 “매출/이익 분배”를 직접 제공하는 구조(배당주형)는  
>   한국 규제상 **증권형 토큰(STO)** 이슈가 매우 강해지므로  
>   **초기 버전에서는 채택하지 않는다.**  
> - 이 문서에서는 **소비자 혜택/멤버십/경험 중심**의 설계에 집중한다.

## 5. 정책 및 규제 방향 (고레벨)

### 5.1 Non-goals (하지 않는 것)

- NFT를 **일반 결제수단**으로 사용하지 않는다.
  - 투어/상품 결제는 원화/카드/내부 포인트를 사용
- NFT를 **투자상품/배당상품**으로 홍보하지 않는다.
  - “수익 보장, 수익 분배, 배당” 등의 표현 금지
- 플랫폼이 **암호화폐 거래소(VASP)** 역할을 하지 않는다.
  - 입출금, 환전, 수탁 기능 등을 직접 제공하지 않음
  - 2차 거래는 OpenSea 등 외부 마켓에 위임

### 5.2 방향성

- **NFT = 신뢰/평판/경험의 상징**  
- 가격 형성/희소가치는 **커뮤니티와 시장이 자연스럽게 만드는 것**  
- 플랫폼은:
  - 룰 제공자 / 심판 / 인프라 제공자 역할에 집중

## 6. 기술 설계 요약 (개발자/투자자용)

### 6.1 전체 아키텍처 (텍스트 다이어그램)

- **온체인 영역**
  - 블록체인 (예: Polygon)
  - NFT 스마트컨트랙트 (ERC-1155 기반 Creator Membership)
  - 메타데이터 서버 (Dynamic NFT를 위한 metadata endpoint)

- **오프체인 영역 (Replit 기반)**
  - 백엔드 API 서버 (Node.js/TypeScript 등)
  - 데이터베이스 (예: Supabase/PostgreSQL)
  - 웹 프론트엔드 (React/Next 등, Replit에서 호스팅)
  - 외부 연동:
    - 블록체인 노드 서비스 (Alchemy/Infura 등)
    - OpenSea 컬렉션 링크

### 6.2 주요 데이터 모델 (초안)

(실제 DB 스키마는 개발 단계에서 세부 조정 예정)

1. `users`
   - `id`
   - `email`
   - `name`
   - `role` (user/creator/admin)
   - `created_at`

2. `wallets`
   - `id`
   - `user_id` (FK → users)
   - `address` (지갑 주소)
   - `chain` (예: polygon)

3. `creators`
   - `id`
   - `user_id`
   - `trust_score` (신뢰도 점수)
   - `is_eligible_for_nft` (발행 자격 여부)
   - `created_at`

4. `nft_collections`
   - `id`
   - `creator_id`
   - `name`
   - `season`
   - `supply_cap`
   - `mint_start_at`
   - `mint_end_at`
   - `contract_address`
   - `token_id` (ERC-1155 기준)
   - `status` (draft/approved/active/closed)

5. `nft_benefit_profiles`
   - `id`
   - `collection_id`
   - `name`
   - `tier_rules` (JSON: 예. 1개=bronze,5개=silver,10개=legend)
   - `benefits` (JSON: 티어별 혜택 정의)

6. (옵션) `user_nft_snapshots`
   - `user_id`
   - `wallet_address`
   - `collection_id`
   - `balance`
   - `tier`
   - `snapshot_at`

> 실제 NFT 보유 여부/수량은 **온체인**이 진짜 근거이고,  
> `user_nft_snapshots`는 캐시/분석용으로 사용.

### 6.3 핵심 API (초안)

| Method | Path                                 | 설명 |
|--------|--------------------------------------|------|
| GET    | `/me`                                | 내 계정 정보 조회 |
| POST   | `/wallets/connect`                   | 지갑 주소 등록/연결 |
| GET    | `/me/nft/collections`                | 내가 관련된 컬렉션(홀더/크리에이터) 목록 |
| GET    | `/me/nft/status`                     | 지갑 기준 현재 NFT 보유 수량, 티어, 혜택 요약 |
| POST   | `/creator/nft/collections`           | (크리에이터용) NFT 컬렉션 발행 신청 |
| GET    | `/creator/nft/collections`           | (크리에이터용) 내 컬렉션 목록/상태 조회 |
| POST   | `/creator/nft/collections/{id}/submit` | 발행 수수료 결제 후, 실제 온체인 발행 요청 |
| GET    | `/collections/{id}`                  | 컬렉션 상세 정보 (공개용) |
| GET    | `/collections/{id}/benefits`         | 컬렉션 혜택/티어 구조 조회 |
| GET    | `/collections/{id}/opensea-link`     | OpenSea 컬렉션/아이템 링크 반환 |

- (1) 투어게더 내부에서 간단한 민팅 기능 제공, 또는

## 7. 단계별 개발 로드맵 (초안)

### Phase 1 – 내부 PoC (기술 검증)

- 단일 크리에이터 + 단일 컬렉션으로 파일럿
- 기능:
  - 지갑 연결
  - 온체인 NFT 보유 수량 조회
  - 티어 계산 및 화면 표시
- OpenSea 연동: 컬렉션 링크만 사용

### Phase 2 – 크리에이터 클로즈드 베타

- 실제 투어게더 가이드/셀러 중 3~10명 선정
- 각자 전용 컬렉션 1개씩 발행
- 기본 혜택(우선 예약, 비밀 지도, 소규모 이벤트) 적용
- 운영 데이터/피드백 수집

### Phase 3 – 정식 기능화

- 크리에이터 대상을 단계적으로 확대
- 발행 심사/수수료/혜택 템플릿 고도화
- Dynamic NFT(시간/거래 횟수 기반 외관 변화) 도입
- 분석 대시보드(홀더 분포, 재거래율 등) 제공
